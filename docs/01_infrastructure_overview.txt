1. EXECUTIVE SUMMARY
================================================================================

Nexus Technology Solutions operates a hybrid IT infrastructure spanning three
on-premises data centers and a multi-cloud environment across Microsoft Azure
and Amazon Web Services. The infrastructure supports approximately 340 internal
employees, 87 active managed service customers, and an average of 2.1 million
API transactions per day across our customer-facing platforms.

Total infrastructure spend for fiscal year 2025 was $6.4 million, split
approximately 55% on-premises and 45% cloud. The infrastructure team consists
of 18 engineers organized into four functional groups: compute and
virtualization, networking, storage and backup, and cloud operations.

Current infrastructure availability SLA is 99.95% for Tier 1 services and
99.9% for Tier 2 services, measured monthly. Actual availability for calendar
year 2025 was 99.97% for Tier 1 and 99.91% for Tier 2.

3. COMPUTE INFRASTRUCTURE
================================================================================

3.1 VIRTUALIZATION PLATFORM
-----------------------------
Nexus operates VMware vSphere 8.0 Update 2 as the primary hypervisor platform
across all three physical locations. The vSphere environment is managed through
a centralized vCenter Server instance hosted in DC-CHI with a secondary
vCenter in DC-DAL for failover.

VMware vSAN 8.0 is deployed in DC-CHI for hyper-converged storage on the
primary compute cluster. DC-DAL uses traditional SAN-attached storage.

vSphere Cluster Summary:
- DC-CHI Cluster A (Production): 12x Dell PowerEdge R750, 2x Intel Xeon Gold
  6338 (64 cores total), 1.5TB RAM per host, 25GbE networking
- DC-CHI Cluster B (Development/Test): 6x Dell PowerEdge R640, 2x Intel Xeon
  Silver 4316 (40 cores total), 512GB RAM per host, 10GbE networking
- DC-DAL Cluster A (DR): 8x Dell PowerEdge R750, 2x Intel Xeon Gold 6338,
  1TB RAM per host, 25GbE networking
- EDGE-NYC (Edge): 2x Dell PowerEdge R440, 1x Intel Xeon Gold 6226R,
  256GB RAM per host, 10GbE networking

Total vSphere Inventory: 28 physical hosts, 847 virtual machines
Average vCPU Overcommit Ratio: 4.2:1
Average RAM Overcommit Ratio: 1.3:1

3.2 KUBERNETES PLATFORM
------------------------
Nexus operates Rancher Kubernetes Engine 2 (RKE2) as the container
orchestration platform for all containerized workloads. There are currently
four distinct Kubernetes clusters:

Production Cluster (k8s-prod):
- 6 control plane nodes (3 per data center for HA)
- 24 worker nodes in DC-CHI, 12 worker nodes in DC-DAL
- Node spec: 32 vCPU, 128GB RAM per node
- Current workloads: 156 namespaces, 892 running pods
- Storage: Longhorn distributed storage with Ceph backend

Development Cluster (k8s-dev):
- 3 control plane nodes (all DC-CHI)
- 8 worker nodes
- Node spec: 16 vCPU, 64GB RAM per node

QA Cluster (k8s-qa):
- 3 control plane nodes (all DC-CHI)
- 6 worker nodes
- Node spec: 16 vCPU, 64GB RAM per node

Edge Cluster (k8s-edge):
- 2 worker nodes (EDGE-NYC), no dedicated control plane
- Managed as a downstream cluster from k8s-prod control plane
- Hosts caching and CDN offload workloads only

Kubernetes toolchain:
- Helm 3.14 for package management
- ArgoCD 2.10 for GitOps deployments
- Prometheus + Grafana for cluster monitoring
- Falco for runtime security
- Cert-Manager for TLS certificate management
- Istio 1.20 service mesh on k8s-prod

5. NETWORKING ARCHITECTURE
================================================================================

5.1 WAN CONNECTIVITY
---------------------
Nexus operates a private MPLS WAN provisioned through AT&T Business:

DC-CHI (Primary):
- Primary circuit: 10Gbps MPLS (AT&T)
- Secondary circuit: 1Gbps dedicated internet (Comcast Business)
- DIA internet: 10Gbps (Comcast Business)

DC-DAL:
- MPLS circuit: 1Gbps (AT&T)
- DIA internet: 1Gbps (Zayo)

EDGE-NYC:
- Internet only: 1Gbps (Zendcom via Equinix IX)

WAN traffic is managed by Cisco SD-WAN (Viptela) for intelligent path
selection and application-aware routing. Critical applications are pinned
to MPLS. Internet-destined traffic breaks out locally at each site.

5.2 LAN ARCHITECTURE
---------------------
Core switching at DC-CHI follows a spine-leaf design:
- 2x Cisco Nexus 93600CD-GX spine switches (100GbE uplinks)
- 8x Cisco Nexus 93180YC-FX3 leaf switches (25GbE server ports)
- Dedicated out-of-band management network (Cisco Nexus 3172)

VLANs in use at DC-CHI:
- VLAN 10: Production servers
- VLAN 20: Development/Test servers
- VLAN 30: Management/OOB
- VLAN 40: Storage (iSCSI)
- VLAN 50: VMware vMotion
- VLAN 60: Kubernetes pod network (overlay via Istio)
- VLAN 70: Corporate end-user devices
- VLAN 100: DMZ / customer-facing services

5.3 FIREWALL AND SECURITY
--------------------------
Palo Alto PA-5250 firewalls operate in active-passive HA at DC-CHI with
Panorama 11.0 centralized management. Firewalls provide:
- Layer 7 application inspection
- Threat Prevention subscription (IPS/IDS)
- URL Filtering
- WildFire sandboxing for malware analysis
- GlobalProtect VPN for remote access (6,200 licensed users)

VPN gateway capacity: 10,000 concurrent sessions
Current peak VPN usage: 287 concurrent sessions (end of business Tuesday)

Palo Alto Prisma Access is deployed for ZTNA to Kubernetes workloads,
replacing legacy VPN access for developer SSH access to production systems.

5.4 DNS AND DHCP
-----------------

- DHCP: Infoblox DDI, 12 scopes, approximately 2,400 active leases

7.2 DISASTER RECOVERY PROCEDURES
----------------------------------
Full DR procedures are documented in the Business Continuity Plan (BCP-2025)
maintained by the Infrastructure team and reviewed quarterly. Key steps:

1. Incident commander declares DR event after confirming DC-CHI is unavailable
   for more than 30 minutes with no estimated restoration time
2. On-call infrastructure lead contacts all Tier 1 application owners
3. Veeam DR orchestration job initiated for DC-CHI VM replication to DC-DAL
4. DNS failover executed via Cloudflare API (automated script: dr-failover.sh)
5. Azure Front Door origin group switched from on-prem to Azure-only
6. Customer communication sent from Statuspage.io within 15 minutes of DR
   declaration
7. Estimated total failover time for Tier 1 services: 3.5 hours

8.1 MONITORING STACK
---------------------
Nexus operates a full-stack observability platform built on open-source
components deployed in Kubernetes:

Metrics:
- Prometheus 2.49 (collection and storage)
- Thanos (long-term metrics storage, 2-year retention)
- Grafana 10.3 (visualization, 200+ dashboards)
- AlertManager (alert routing and deduplication)

Logs:
- Fluent Bit (log shipping from all nodes and pods)
- Loki 3.0 (log aggregation and indexing)
- Grafana (log querying via Loki datasource)
- Additionally: all logs forwarded to Azure Log Analytics

Traces:
- OpenTelemetry Collector (trace ingestion)
- Tempo 2.3 (trace storage)
- Grafana (trace visualization)

Synthetic Monitoring:
- Grafana k6 Cloud (external synthetic monitoring from 8 global locations)
- Checks every 60 seconds for all Tier 1 endpoints
- Uptime Robot (secondary external monitoring for customer-facing URLs)

8.2 ALERTING STRUCTURE
------------------------
Alert routing is managed through AlertManager with PagerDuty integration:

P1 (Critical):
- Definition: Complete service unavailability, data loss risk, security breach
- Response time: 15 minutes
- Notification: PagerDuty (immediate call), Slack #incidents, SMS to on-call
- On-call rotation: Infrastructure lead + application lead (weekly rotation)

P2 (High):
- Definition: Degraded performance, partial service impact, capacity threshold
- Response time: 1 hour
- Notification: PagerDuty (30-min delay), Slack #alerts

P3 (Medium):
- Definition: Non-customer-impacting issues, warning thresholds
- Response time: Next business day
- Notification: Slack #monitoring only

Current alert volume (last 30 days):
- P1 alerts fired: 3 (all resolved within SLA)
- P2 alerts fired: 47
- P3 alerts fired: 312
